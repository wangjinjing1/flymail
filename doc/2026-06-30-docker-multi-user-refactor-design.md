# FlyMail Docker 多用户重构设计

日期：2026-06-30

## 1. 目标

将项目从“飞牛应用中心安装版”重构为“独立 Docker 多用户版”，去除飞牛运行环境依赖，同时保留现有邮件能力与历史同步能力。

本次重构目标：

1. 建立 FlyMail 自有用户体系，不再依赖飞牛请求头识别用户
2. 支持管理员与普通用户两类角色
3. 管理员账号由 `.env` 初始化
4. 普通用户可以修改自己的密码
5. 管理员可以创建用户、重置密码、启用或禁用用户，但不能查看原密码
6. 保留多邮箱、历史同步、断点续传、附件/图片缓存、通知等能力
7. 清理飞牛专属安装包与无关内容
8. 按当前 Docker 版本重写项目文档

## 2. 非目标

本次不处理以下内容：

1. 不提供明文密码可见能力
2. 不接入第三方单点登录
3. 不支持邮箱账号跨用户共享
4. 不引入复杂权限模型，仅保留 `admin` 与 `user`
5. 不做覆盖所有历史安装形态的自动迁移工具

## 3. 现状判断

### 3.1 可复用能力

项目原有以下能力可以继续复用：

1. 账号、邮件缓存、通知、签名、历史同步等业务模型已经围绕 `user_uid` 做隔离
2. 历史邮件同步已经具备后台拉取、进度查询、暂停/继续、断点恢复、附件与图片下载能力
3. Docker 化、MySQL 接入、本地 `data/` 目录持久化已经具备基础形态

### 3.2 主要问题

重构前的核心问题：

1. 用户身份来自飞牛环境注入的请求头
2. `/api/user` 等接口语义与飞牛用户体系绑定
3. WebSocket 与多处业务路由依赖飞牛上下文
4. README、安装说明、注释中残留飞牛语义
5. 仓库内存在飞牛专属目录与包装内容

## 4. 总体方案

采用“保留业务能力，替换身份体系”的重构策略：

1. 保留现有邮件与同步业务模块
2. 新建本地用户模型与会话认证
3. 将 `user_uid` 的语义切换为 FlyMail 本地用户 ID
4. 所有接口统一从登录态获取当前用户
5. 删除飞牛专属目录与打包残留

这样可以避免重写整套邮件业务，同时彻底去掉平台依赖。

## 5. 数据模型设计

### 5.1 新增 `users` 表

建议字段：

- `id`：UUID 主键
- `username`：唯一用户名
- `password_hash`：密码哈希
- `role`：`admin` / `user`
- `status`：`active` / `disabled`
- `created_at`
- `updated_at`

### 5.2 现有 `user_uid` 字段处理

现有业务表中的 `user_uid` 字段保留，不统一改名，避免大范围 SQL 与代码重命名。

语义更新为：

- `user_uid == users.id`

主要涉及：

- `accounts`
- `cached_messages`
- `notifications`
- `user_settings`
- `signatures`
- `history_sync_jobs`

### 5.3 管理员初始化

服务启动时检查以下环境变量：

- `FLYMAIL_ADMIN_USERNAME`
- `FLYMAIL_ADMIN_PASSWORD`
- `FLYMAIL_SESSION_SECRET`

启动逻辑：

1. 不存在管理员时自动创建
2. 存在同名管理员时不覆盖已有密码
3. 缺少必要环境变量时直接报错并阻止启动

## 6. 鉴权设计

### 6.1 认证方式

使用基于 HTTP Only Cookie 的会话认证：

1. 更适合当前浏览器内网场景
2. 前端改动成本较低
3. 登录、登出、当前用户状态更容易统一处理

### 6.2 新增接口

认证接口：

1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/me`
4. `POST /api/auth/change-password`

管理员接口：

1. `GET /api/admin/users`
2. `POST /api/admin/users`
3. `POST /api/admin/users/{id}/reset-password`
4. `POST /api/admin/users/{id}/toggle-status`

### 6.3 后端依赖调整

原有通过飞牛请求头取用户的依赖改为从登录态取用户。

新增公共依赖：

1. `get_current_user()`
2. `get_current_user_id()`
3. `require_admin()`

所有原本依赖飞牛用户头的路由、WebSocket、后台任务入口统一切换到本地登录态。

## 7. 权限设计

### 7.1 普通用户

普通用户可以：

1. 登录 / 登出
2. 修改自己的密码
3. 管理自己的邮箱账号
4. 查看自己的邮件、通知、签名、设置、历史同步任务

普通用户不能：

1. 查看其他用户数据
2. 访问用户管理页面
3. 创建、禁用、重置其他用户

### 7.2 管理员

管理员可以：

1. 登录 / 登出
2. 修改自己的密码
3. 创建用户
4. 重置用户密码
5. 启用 / 禁用用户
6. 查看用户列表和基础信息

管理员不能：

1. 查看任何用户的原始密码

## 8. 前端设计

### 8.1 登录流

新增登录页，包含：

1. 用户名
2. 密码
3. 登录按钮

前端启动时先请求 `/api/auth/me`：

1. 已登录则进入主应用
2. 未登录则跳转登录页

### 8.2 导航调整

新增或调整以下入口：

1. 当前用户信息
2. 登出入口
3. 修改密码入口
4. 仅管理员可见的“用户管理”
5. 历史同步进度页

### 8.3 用户管理页

仅管理员可见，支持：

1. 用户列表
2. 创建用户
3. 重置密码
4. 启用 / 禁用

页面不显示明文密码，也不回显密码哈希。

## 9. 文档重写范围

README 需要覆盖：

1. 项目简介
2. 当前功能列表
3. Docker 启动方式
4. `.env` 配置说明
5. MySQL 连接说明
6. `data/` 目录映射与用途
7. 多用户与管理员权限
8. 历史邮件同步说明
9. 原作者致谢

## 10. 飞牛残留清理范围

需要删除或替换：

1. `flymail/` 目录
2. 飞牛应用中心安装说明
3. 飞牛专属文档
4. 飞牛请求头依赖
5. 飞牛用户语义相关注释与描述

可以保留但调整语义：

1. 现有数据库字段名 `user_uid`
2. 少量内部变量名

原则是优先保证当前行为正确，而不是为了命名洁癖做大面积重命名。

## 11. 实施阶段

### 阶段 1：认证与用户基础

1. 增加 `users` 表
2. 增加密码哈希与会话逻辑
3. 增加管理员初始化
4. 改造依赖注入与 `/api/user`

### 阶段 2：业务接口接管

1. 替换业务路由对飞牛头的依赖
2. 替换 WebSocket 对飞牛头的依赖
3. 确保历史同步、通知、签名、账号、邮件等都从登录态取用户

### 阶段 3：前端登录与用户管理

1. 登录页
2. 当前用户状态管理
3. 修改密码
4. 管理员用户管理页

### 阶段 4：清理与文档

1. 删除飞牛残留目录与无关内容
2. 重写 README
3. 校验 `.gitignore`
4. 验证 Docker 部署说明

## 12. 验证标准

做到以下结果才算完成：

1. 未登录不能访问主业务页面
2. 管理员可使用 `.env` 初始化账号登录
3. 管理员可创建普通用户
4. 普通用户登录后只能看到自己的数据
5. 普通用户可以修改自己的密码
6. 管理员可以重置用户密码，但不能查看原密码
7. 历史邮件同步在新登录体系下仍可工作
8. 前端构建通过
9. 后端编译或导入检查通过
10. 文档与当前 Docker 用法一致

## 13. 风险与约束

1. 原仓库部分中文文件存在编码问题，重构时需要顺手修复
2. `user_uid` 使用范围很广，不适合做激进重命名
3. WebSocket 需要在改造后继续稳定推送通知与同步状态
4. 旧数据如果来自飞牛环境，迁移时需要接受一次语义切换

## 14. 实施策略

采用“先接管身份，再清理环境”的顺序：

1. 先建立本地用户模型与登录流
2. 再切换所有业务依赖
3. 最后清理飞牛残留并统一文档

这样更容易保证过程中的功能可验证性与回归可控性。
