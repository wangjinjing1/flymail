# FlyMail Docker 多用户全量重构设计

日期：2026-06-30

## 1. 目标

将当前项目从“飞牛应用中心安装版”彻底重构为“原生 Docker 多用户邮件系统”，去除飞牛网关和飞牛用户体系依赖，同时保留现有邮件功能与历史同步能力。

本次重构目标：

1. 建立 FlyMail 自有用户体系，不再依赖飞牛请求头识别用户。
2. 支持管理员和普通用户两类角色。
3. 管理员账号由 `.env` 初始化。
4. 普通用户允许自行修改密码。
5. 管理员允许创建用户、重置密码、禁用/启用用户，但不能查看原密码。
6. 保留现有多邮箱、历史邮件同步、断点续传、附件/图片缓存、通知等能力。
7. 删除飞牛专属安装包和无关内容。
8. 重写 README，按当前 Docker 版本交付说明。

## 2. 非目标

本次不做以下内容：

1. 不实现密码明文可见。
2. 不做第三方单点登录。
3. 不做邮件账号跨用户共享。
4. 不做复杂权限模型，只有 `admin` 和 `user` 两种角色。
5. 不做数据库无损自动迁移脚本覆盖所有历史环境；以当前仓库版本向 Docker 版演进为主。

## 3. 当前状态判断

### 3.1 已有能力

当前项目已经具备以下可复用能力：

1. 邮件账号、缓存邮件、通知、签名、历史同步任务等数据都通过 `user_uid` 做隔离。
2. 历史邮件同步已经支持：
   - 后台慢慢拉取
   - 同步进度查看
   - 暂停 / 继续
   - 从断点继续
   - 下载附件和内嵌图片
3. Docker 化、MySQL 连接、`data/` 本地目录存储已经完成基础改造。

### 3.2 当前阻塞点

当前项目仍然深度依赖飞牛环境：

1. 用户身份来自 `X-Trim-Userid` / `X-Trim-Username` 请求头。
2. `/api/user` 返回飞牛用户信息。
3. WebSocket 和多数业务路由通过飞牛头识别当前用户。
4. 代码注释、README、安装说明、网关前缀逻辑仍然带有飞牛语义。
5. 仓库内仍存在 `flymail/` 应用包目录等飞牛专属内容。

## 4. 总体方案

采用“全量重构型”方案：

1. 保留现有业务模块和数据隔离结构。
2. 重建本地用户域模型。
3. 将 `user_uid` 的语义从“飞牛用户 ID”改为“FlyMail 本地用户 ID”。
4. 所有接口统一从登录态获取当前用户。
5. 删除飞牛应用包与无关飞牛内容。

这样可以避免重写所有业务表结构，同时彻底去掉飞牛依赖。

## 5. 数据模型设计

### 5.1 新增 users 表

新增 `users` 表，字段建议如下：

- `id`：字符串主键，UUID
- `username`：用户名，唯一
- `password_hash`：密码哈希
- `role`：角色，`admin` / `user`
- `status`：状态，`active` / `disabled`
- `created_at`
- `updated_at`

### 5.2 现有 user_uid 字段处理

现有表中的 `user_uid` 字段保留，不统一改列名，避免大范围 SQL 改造，但语义更新为：

- `user_uid == users.id`

涉及表包括但不限于：

1. `accounts`
2. `cached_messages`
3. `notifications`
4. `user_settings`
5. `signatures`
6. `history_sync_jobs`

### 5.3 管理员初始化

服务启动时检查 `.env`：

- `FLYMAIL_ADMIN_USERNAME`
- `FLYMAIL_ADMIN_PASSWORD`
- `FLYMAIL_SESSION_SECRET`

启动逻辑：

1. 若不存在管理员用户，则创建。
2. 若存在同名管理员，则不覆盖密码。
3. 若缺少必要环境变量，启动时给出明确错误。

## 6. 鉴权设计

### 6.1 认证方式

采用项目自有会话鉴权。

建议使用 HTTP Only Cookie 会话，而不是让前端自己长期保存明文 token。原因：

1. 更适合浏览器内网应用。
2. 前端改造成本更低。
3. 便于统一处理登录、登出和当前用户状态。

### 6.2 新增接口

新增认证接口：

1. `POST /api/auth/login`
2. `POST /api/auth/logout`
3. `GET /api/auth/me`
4. `POST /api/auth/change-password`

管理员新增接口：

1. `GET /api/admin/users`
2. `POST /api/admin/users`
3. `POST /api/admin/users/{id}/reset-password`
4. `POST /api/admin/users/{id}/toggle-status`

### 6.3 后端依赖改造

现有 `deps.get_uid()` 将被重写为从登录态取当前用户 ID。

新增依赖函数：

1. `get_current_user()`
2. `get_current_user_id()`
3. `require_admin()`

所有原先依赖飞牛头的路由、WebSocket、后台任务入口统一改为依赖本地登录态。

## 7. 权限设计

### 7.1 普通用户

普通用户可以：

1. 登录 / 登出
2. 修改自己的密码
3. 管理自己的邮箱账号
4. 查看自己的邮件、通知、签名、设置、历史同步任务

普通用户不可以：

1. 查看其他用户数据
2. 访问用户管理页面
3. 创建或禁用其他用户

### 7.2 管理员

管理员可以：

1. 登录 / 登出
2. 修改自己的密码
3. 创建用户
4. 重置用户密码
5. 启用 / 禁用用户
6. 查看用户列表和基础信息

管理员不可以：

1. 查看任何用户的原密码

## 8. 前端设计

### 8.1 登录流

新增登录页：

1. 用户名
2. 密码
3. 登录按钮

前端启动时先请求 `/api/auth/me`：

1. 已登录则进入主应用
2. 未登录则跳转登录页

### 8.2 导航变化

新增或调整：

1. 右上角或侧边栏显示当前用户
2. 登出入口
3. 修改密码入口
4. 管理员专属“用户管理”

### 8.3 用户管理页面

仅管理员可见，支持：

1. 用户列表
2. 创建用户
3. 重置密码
4. 启用 / 禁用

页面不显示密码原文，也不回显密码哈希。

## 12. 验证标准

做到以下结果才算完成：

1. 未登录不能访问主业务页面。
2. 管理员可用 `.env` 初始账号登录。
3. 管理员可以创建普通用户。
4. 普通用户登录后只能看到自己的数据。
5. 普通用户可以修改自己的密码。
6. 管理员可以重置用户密码，但不能查看原密码。
7. 历史邮件同步功能在登录体系切换后仍可正常工作。
8. 前端构建通过。
9. 后端编译或导入检查通过。
10. README 与当前 Docker 使用方式一致。

## 13. 风险与约束

1. 当前仓库存在部分中文乱码文件，README 和若干注释可能需要顺带修复编码问题。
2. 现有代码使用 `user_uid` 很广，不能贸然做列名重命名，否则改动面过大。
3. WebSocket 现在也依赖飞牛头，改造时需要确保通知和同步状态推送不断。
4. 旧数据若来自飞牛环境，可能只有 `default` 或飞牛用户 ID，需要接受一次性语义切换。

## 14. 实施策略

本次采用“先接管身份，再清理环境”的顺序：

1. 先把登录和用户模型接起来。
2. 再把所有业务依赖切过去。
3. 最后删除飞牛残留并重写 README。

这样更容易保持过程中的功能可验证性。
